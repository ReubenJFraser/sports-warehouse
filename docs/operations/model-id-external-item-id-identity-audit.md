# Model ID vs external_item_id Identity Audit (Documentation-Only)

Date: 2026-05-26  
Scope: naming-layer audit only (no DB/data/code mutation)

## Purpose

This audit documents the current naming-layer relationship between:

- ProductDB `model_id` (logical/canonical identity name)
- MySQL `item.external_item_id` (current physical storage column used in operations scripts)
- MySQL `item.db_itemId`
- MySQL `item.itemId`

## Findings by question

### 1) What does current documentation say `model_id` is for?

Current contracts consistently define `model_id` as the canonical logical product identity key:

- Generated by governed structured rules (Contract 19).
- Canonical bridge across ProductDB, MySQL row matching, and image matching (Contract 22).
- Flat canonical identity string; runtime/storage paths may use governed implementation variants (Contract 23).
- Canonical product identity anchor for Ryderwear folder governance (Contract 24).

### 2) Where does `external_item_id` appear in schema/docs/code?

`external_item_id` appears in operational SQL and importer tooling as an actual MySQL-side field used for identity joins/mapping:

- SQL plans and checks explicitly join `item.external_item_id = product_import_staging.model_id`.
- SQL migration/readiness scripts explicitly discuss widening `item.external_item_id` for `model_id` values.
- Import diagnostics reference `external_item_id` alongside `db_itemId`.

### 3) Does existing documentation explain why MySQL column is `external_item_id` instead of `model_id`?

No explicit architectural rationale document was found that formally explains the naming difference.

Observed state is pragmatic/operational: governance uses `model_id` as canonical logical name, while existing MySQL physical schema and scripts currently use `external_item_id` as the storage/join column for that identity value.

### 4) Is `external_item_id` currently functioning as the physical MySQL storage column for ProductDB `model_id`?

Yes, based on current operational SQL evidence:

- insert/update/readiness scripts repeatedly map `pis.model_id` to `item.external_item_id`
- collision checks and matching logic are built on that join

Therefore for local MySQL execution targets in this workflow, `external_item_id` is the active physical column to match ProductDB `model_id` values.

### 5) What is `db_itemId` for, and how is it different from `itemId` and `external_item_id`?

Current docs indicate:

- `item.itemId`: primary key of MySQL `item` rows (system PK).
- `item.db_itemId`: secondary/business linkage identifier used by some import/reconciliation paths and staged insert planning in this repo’s current operations docs.
- `item.external_item_id`: external canonical identity string column currently used to store/match ProductDB `model_id` values.

In short:

- `itemId` = row PK identity
- `db_itemId` = operational linkage identifier used in CSV/import workflows
- `external_item_id` = physical string column used for canonical model identity matching from ProductDB `model_id`

### 6) What naming convention should future scripts use?

For current local MySQL compatibility:

- Treat `model_id` as the canonical logical identity name in docs/planning/ProductDB.
- Use `item.external_item_id` in executable SQL against current local MySQL schema when matching that identity.
- Be explicit in script comments, e.g. `-- ProductDB model_id maps to item.external_item_id in local MySQL`.

### 7) Rename now or later?

Do **not** rename now.

Any rename from `external_item_id` to `model_id` should be a separate, controlled schema-normalization + compatibility migration project with importer/runtime impact review.

That rename is out of scope for Ryderwear Batch 2 image-update operations and should not be coupled to current SQL regeneration.

## Notes on schema evidence quality

- `db/sportswh_dump.sql` is a historical dump and appears to predate current local DBeaver-observed schema state for `item` (it does not present `external_item_id`/`db_itemId` in the shown `CREATE TABLE item` block).
- Current operations SQL/docs and local DBeaver verification are aligned that local MySQL currently includes `external_item_id` and uses it in active workflows.

## Final conclusion

Confirmed conclusion (matching expected direction):

- `model_id` is the canonical **logical** identity name in ProductDB/governance.
- `external_item_id` is the current MySQL **physical** column carrying that identity value.
- SQL targeting current local MySQL must use `item.external_item_id` when matching ProductDB `model_id` values.
- Do not rename columns during Ryderwear image update workflow.
- Any future rename to `model_id` is a separate schema-normalization migration project.
