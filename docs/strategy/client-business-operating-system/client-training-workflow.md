# Client training workflow

The Client Business Operating System should have two clear workflow tiers. Most small-business clients should use the client-friendly workflow. The assisted advanced workflow should remain primarily for the developer, or for an unusually technical client who wants deeper involvement.

## Tier 1: Client-friendly workflow

This is what most small-business clients should use day to day. It should be safe, guided, and practical.

Clients can be trained to:

* Log in to admin.
* Use the admin dashboard to inspect products, services, images, and content status.
* Edit products or services through guided forms.
* Upload or approve images.
* Review content drafts before they are published.
* Use guided forms instead of editing code or database records directly.
* Approve or defer changes.
* Inspect readiness reports and understand what is ready, incomplete, or blocked.
* Ask ChatGPT for help writing descriptions, product updates, service explanations, notices, or customer-facing messages.
* Contact the developer for technical changes, new features, design changes, deployment questions, or anything that feels risky.

The client-friendly workflow should avoid unnecessary technical burden. The client should not need to understand branches, commits, deployment, databases, manifests, or code to get value from the system.

## Tier 2: Assisted advanced workflow

This is what the developer may use, or what an unusually technical client may learn with explicit training and support.

The assisted advanced workflow may include:

* ChatGPT Premium for deeper content drafting, planning, and support.
* Codex for developer-assisted code, documentation, and repository work.
* GitHub for version-controlled changes, issues, pull requests, review history, and project records.
* Version-controlled changes for safer technical updates.
* Database/admin tooling for structured content and operational review.
* Deployment workflow for controlled releases.
* Documentation and training for repeatable maintenance.

Most clients should not be expected to learn PowerShell, GitHub, branches, commits, pull requests, database tooling, deployment, or local developer setup as a default requirement. Those belong mainly to the developer-assisted workflow.

## Training principle

The Website + Business Admin System should make everyday business updates feel approachable. The client-friendly workflow should focus on visible business tasks: update a service, approve a product image manifest result, review approvals, inspect safe publishing gates, and ask for help when unsure. The assisted advanced workflow should protect technical complexity behind developer support.
